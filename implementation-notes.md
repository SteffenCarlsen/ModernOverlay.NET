# Implementation Notes

## Design Decisions

- The initial implementation targets the spec's Phase 0 and early Phase 1 architecture slice: solution layout, central package management, nullable/analyzer settings, CI, public API skeleton, Win32 style/lifecycle foundation, Direct2D dependency proof, smoke tests, and documentation artifacts.
- `ModernOverlay` is the public facade and currently references `ModernOverlay.Win32` directly so `OverlayWindow.CreateAsync` is usable immediately. A later backend composition layer can invert this dependency if the API surface stabilizes around multiple windowing hosts.
- The first `DrawContext` is a no-op command surface with validation for clip/transform stack balance. This lets samples and tests compile against the intended immediate-mode API before the Direct2D command sink is implemented.
- `FrameRateLimit.DisplayDefault` resolves the overlay window's current monitor refresh rate through Win32 display queries when the frame loop starts, falling back to 60 FPS if Windows does not report a plausible value.
- `FrameRateLimit` now matches the spec shape more closely as a readonly record struct with fractional `Fixed(double)` frame-rate support.
- MSTest is used for the first test project because the installed .NET skill guidance recommends the modern `MSTest` metapackage for new projects.
- HWND creation, style mutation, destruction, and render callbacks now marshal through a dedicated Win32 owner thread. The owner thread initializes COM and waits with `MsgWaitForMultipleObjectsEx` on a work signal plus the Windows message queue.
- `OverlayWindow.RunAsync` now runs the frame loop on the Win32 owner thread. Fixed-rate scheduling uses a native waitable timer in the owner-thread wait set alongside queued work, cancellation, and window messages; unlimited mode runs continuously while still draining queued work and messages between frames.
- `TransparencyMode.Auto` currently maps to full-client `DwmExtendFrameIntoClientArea`. That is a provisional bootstrap default and must still be validated visually with the real Direct2D renderer.
- The rendering layer now has internal backend abstractions (`IRenderBackend`, `IDrawCommandSink`, backend generation, frame info, begin/end frame results). `OverlayWindow` initializes and drives a `NullRenderBackend` on the owner thread so backend lifecycle is part of the real host path before default Direct2D backend selection is wired into the facade.
- `DrawContext` now forwards draw/fill/clear calls to an internal command sink. This preserves the public immediate-mode surface while creating the command path needed for Direct2D and future command-list/IPC ingestion.
- `DrawContext` now forwards clip and transform state changes to the command sink as well as draw commands. Frame completion fails clearly if a render callback leaves clip or transform scopes active, and the context unwinds renderer state before surfacing the error.
- Overlay-created `DrawContext` instances are now frame-scoped: any retained frame object throws `InvalidOperationException` if used outside the active render callback. Standalone/test draw contexts remain unscoped so low-level command-sink tests and headless API probes can still exercise the immediate-mode surface without an overlay host.
- `ModernOverlay.Direct2D` now creates actual Vortice Direct2D, DirectWrite, and WIC factories plus an `ID2D1HwndRenderTarget` for a real HWND. GPU-backed commands currently include clear, solid/linear-gradient brush line drawing, rectangles, rounded rectangles, ellipses, circles, triangles, descriptor-backed geometry paths with lines plus cubic/quadratic Beziers and elliptical arcs, WIC image drawing with explicit frame-index addressing, source rectangles/opacity/interpolation, text drawing, axis-aligned clips, 3x2 transforms, and reusable stroke styles for dashed/styled outlines.
- A BenchmarkDotNet project is included under `benchmarks/ModernOverlay.Benchmarks` with public API coverage for draw dispatch, clip scopes, text measurement, overlay create/recreate/dispose lifecycle, and Direct2D clear/draw/resize paths. It uses `BenchmarkSwitcher` so filters and dry runs work non-interactively.
- `OverlayResourceManager` now tracks live public resource handles by id/kind/generation/allocation site and can produce a leak report snapshot. Handles unregister deterministically on `Dispose`. Public handles also track native realization snapshots with backend name, backend generation, and native resource kind; Direct2D cache entries register/unregister these snapshots as native resources are created and disposed.
- `BrushHandle` is now the common public base for solid and linear-gradient brushes. Existing solid-brush APIs continue to accept `SolidBrushHandle` through inheritance, while draw/fill/text operations can accept any current brush handle.
- `WindowTarget.FromHwnd` and the first target-tracking path are implemented. An overlay with a target HWND resolves initial bounds from the target and resynchronizes its bounds before rendering.
- `WindowTarget.ByProcessId`, `WindowTarget.ByProcessName`, `WindowTarget.ByTitle`, `WindowTarget.ByWindowTitle`, `WindowTarget.ByClassName`, `WindowTarget.Foreground`/`ForegroundWindow`, and `WindowTarget.FromProvider` now resolve top-level HWNDs through Win32 enumeration, foreground-window APIs, or a user-supplied provider. Provider-backed and discovery-backed targets re-resolve on each tracking pass, which gives basic reacquire behavior for windows that disappear and later return.
- `ByTitle` follows the spec default of ordinal-insensitive contains matching through `MatchMode.Contains`; `ByWindowTitle` remains as an exact-match compatibility helper.
- Target bounds modes now include custom bounds via `OverlayTarget.WithCustomBounds(Func<WindowHandle, WindowBounds?>)`, covering the spec's custom target-bounds hook while keeping the default whole-window/client-area modes intact.
- `Win32WindowQuery` now exposes the required helper surface for validity/visibility, styles, process IDs, foreground/active/desktop/shell windows, owner/child/next/previous relationships, child-window search by class/title, and window/client bounds.
- `Win32WindowZOrder` and `Win32WindowEffects` now expose public helper methods for topmost removal/application, relative placement, DWM frame extension, and DWM blur-behind enable/disable; `Win32OverlayWindow` uses those helpers internally.
- `Win32NativeDiagnostics.LastFailure` records the latest native failure surfaced through `NativeWin32Exception` or `NativeHResultException`, including operation name, Win32 error or HRESULT, and timestamp. The diagnostics sample displays that value as "none" until a native failure is recorded.
- `docs/directcomposition-spike.md` documents the current DirectComposition decision: Direct2D HWND remains the parity backend, `TransparencyMode.DirectComposition` stays unsupported, and DirectComposition must prove transparent composition, click-through/no-activate, resize, device loss, draw parity, benchmark comparison, and acceptable complexity before becoming default.
- The docs set now includes quick start, GameOverlay.NET concept mapping, troubleshooting, transparency validation, DirectComposition decision notes, and a performance guide. These docs intentionally describe the current explicit Direct2D backend registration and current limitations instead of presenting the API as a finished drop-in package.
- The docs set now also includes installation, window modes, target tracking, DPI/multi-monitor behavior, drawing primitives, resource lifetime, device recreation, integration boundary, feature-completeness, and release-validation checklist pages so the spec's non-benchmark deliverables are auditable before performance work resumes.
- `.github/workflows/ci.yml` now defines the Windows CI path for the pinned .NET 11 preview SDK from `global.json`. It restores/builds the solution with binlogs, runs non-integration tests by default, and exposes Windows integration tests as an explicit `workflow_dispatch` option so headless CI does not silently fail on desktop-dependent checks.
- `OverlayZOrder.FollowTarget` now performs a best-effort Win32 `SetWindowPos` call during target tracking using the resolved target HWND as the relative z-order anchor.
- `TargetMinimizedPolicy` separates target-minimized behavior from normal overlay visibility. The default `HideOverlay` policy hides an overlay while the target is iconic and restores it only when the overlay's desired visibility is still true; `PauseRendering` skips render callbacks until the target restores without changing overlay visibility.
- `OverlayWindow.TargetLost` and `OverlayWindow.TargetReacquired` now fire when a previously resolved target disappears and later resolves again. The event args include the configured target descriptor plus the last/resolved HWND and bounds when available, and `OverlayEventSource` emits a separate `TargetReacquired` event.
- `OverlayWindowOptions.TargetTrackingInterval` now throttles target polling when set to a positive duration. The default is 33 ms for GameOverlay.NET parity; callers can set `TimeSpan.Zero` for every-frame tracking.
- `ModernOverlay.Direct2D` now exposes an explicit backend registration hook. `OverlayWindow` uses a registered backend provider when one is present, and the basic sample registers Direct2D before creating its overlay.
- `samples/StickyTargetOverlay` demonstrates target tracking against a project-owned Win32 host window. The sample uses `WindowTarget.FromHwnd`, `TargetTrackingInterval`, and `OverlayZOrder.FollowTarget` while moving the owned target window, which exercises the sticky overlay path without attaching to a third-party process.
- The sample set now includes nine build-validated overlays: basic text, sticky target tracking, shapes/gradients/strokes, reusable text layouts, frame diagnostics, input-mode toggling, hotkey toggling, image drawing, and transparency validation.
- `Measure.Text` now goes through the command sink. The Direct2D backend uses DirectWrite text-layout metrics; the null/no-op path retains the simple estimate for tests and headless fallback behavior.
- `FrameStats` now captures the latest frame count, total frame duration, target interval, actual interval since the previous completed frame, render callback duration, present duration, command count, primitive count, transient text layout count, native resource realization count, current overlay bounds, target HWND, and target bounds when target tracking resolves a window.
- `FrameStats` now also exposes current FPS, average FPS, moving-average frame duration, worst frame duration over the rolling diagnostics window, skipped and dropped frame counts, target-tracking update count, and backend generation. The diagnostics sample displays FPS, frame timing, render thread, command/primitive count, transient text layout count, native resource count, backend name/generation, overlay HWND, target HWND, DPI scale, and last native failure.
- Backend `FrameInfo` now carries frame index, timestamp, elapsed time, delta time, target frame interval, actual frame interval, render thread ID, DPI scale, window bounds, and target bounds.
- `ModernOverlay.Diagnostics.OverlayEventSource` is now wired into the core overlay path. It emits overlay created/destroyed, backend initialized/disposed, device lost/restored, frame rendered with command/primitive/transient-text-layout/native-resource counts, frame over-budget/skipped-frame, excessive text layout creation, target resolved/lost/reacquired, DPI changed, render exception, and resource leak events.
- `OverlayEventSource.BackendFallback` now records backend feature fallbacks. The Direct2D HWND backend emits it once when `PresentMode.VSync` is requested, then falls back to backend-default presentation because `ID2D1HwndRenderTarget` does not expose an explicit VSync mode.
- `ModernOverlay.Diagnostics.OverlayEventSourceLogger` provides the first `Microsoft.Extensions.Logging` adapter by listening to the `ModernOverlay` EventSource and forwarding structured event names/payloads into an `ILogger`.
- Per-monitor DPI awareness is now requested for Win32 overlay windows when `OverlayWindowOptions.DpiMode` is `PerMonitorV2`. The current HWND DPI scale is queried with `GetDpiForWindow`, passed into backend initialization/resizes, included in `FrameStats`, and emitted with frame diagnostics. `WM_DPICHANGED` now applies the suggested bounds, resizes the render backend, raises `BoundsChanged`, and emits a DPI diagnostic event.
- `WindowBounds.FromPixels`, `WindowBounds.FromDips`, `DpiScale.PixelsToDips`, `DpiScale.DipsToPixels`, `OverlayWindow.SetBoundsPixels`, and `OverlayWindow.SetBoundsDips` make the pixel/DIP boundary explicit while preserving the existing physical-pixel `WindowBounds` model.
- `OverlayWindow.Hotkeys` now exposes a first global-hotkey manager. It supports `KeyGesture.CtrlAltO`, A-Z key helpers, F1-F24 helpers, `HotkeyModifiers`, native `RegisterHotKey`/`UnregisterHotKey`, and dispatches `WM_HOTKEY` callbacks through the Win32 owner-thread message pump.
- `OverlayWindow.PointerMoved`, `PointerPressed`, and `PointerReleased` now expose a first interactive pointer surface. Win32 `WM_MOUSEMOVE` and left/right/middle button messages are translated on the owner thread, with public event positions reported in DIPs plus original client pixels.
- `RenderExceptionPolicy` now exposes the spec-named `Continue`, `PauseOverlay`, and `FailFast` policies while keeping `IgnoreAndContinue` as a compatibility alias for `Continue`. `Continue` logs and resumes on the next frame; `PauseOverlay` marks the overlay paused before surfacing the exception; `FailFast` calls `Environment.FailFast` after logging the render exception.
- `OverlayWindowOptions.RejectResourceCreationDuringRender` adds the spec's opt-in render-hot-path guard. When enabled, public resource creation through the overlay's `OverlayResourceManager` throws during the user render callback and leaves no tracked handle behind.
- Drawing helper APIs for `Draw.Box`, `Draw.Crosshair`, and `Draw.Arrow` are implemented by forwarding to existing rectangle/line primitives, so they work across all backends without additional native commands.
- `TextLayoutHandle` can now be drawn and measured through `Draw.Text(layout, ...)` and `Measure.Text(layout)`. The Direct2D backend caches DirectWrite text layouts by handle for reuse.
- `TextLayoutOptions` now gives reusable text layouts explicit maximum width/height, wrapping, horizontal alignment, vertical alignment, and trimming descriptors. The Direct2D backend applies those descriptors to the DirectWrite text format used to create each cached layout.
- `LinearGradientBrushHandle` stores start/end points plus sorted gradient stops. The Direct2D backend realizes those descriptors as `ID2D1LinearGradientBrush` instances using clamp extend mode and standard RGB gamma.
- `IRenderBackend.Recreate` now gives `OverlayWindow.RecreateAsync` a real backend tear-down/reinitialize path. Direct2D disposes factories, render targets, and cached native resources, then reinitializes against the same HWND; null and Direct2D backends both advance backend generation.
- Direct2D now caches native `ID2D1PathGeometry` realizations for reusable `GeometryPath` handles, separately for stroked and filled figure modes. Cached geometries are included in native resource counts, reflected on the public `GeometryPath` handle's native realization snapshots, and disposed on backend recreation.

## Deviations

- The spec lists `Vortice.DirectWrite` and `Vortice.WIC` as direct package references, but those package IDs do not currently resolve from NuGet and no matching assemblies were present through package closure. The implementation pins the resolvable Vortice packages (`Vortice.Direct2D1`, `Vortice.DXGI`, `Vortice.DirectX`, `Vortice.Mathematics`, `Vortice.Win32`) and records DirectWrite/WIC as unresolved spec package assumptions.
- Render scheduling uses a native waitable timer on the owner thread for fixed frame rates. `DisplayDefault` now resolves through `MonitorFromWindow`/`GetMonitorInfo`/`GetDeviceCaps(VREFRESH)` and falls back to a window DC, then to 60 FPS if the platform reports an unusable value.
- Image handles can be created from file paths, byte arrays, read-only memory buffers, or streams; the Direct2D path decodes PNG, JPEG/JPG, and BMP through WIC, converts the requested frame to premultiplied BGRA, and creates an `ID2D1Bitmap` for the active render target. Path-backed images are read into memory before WIC decoding so cached bitmaps do not keep caller files locked. Multi-frame image addressing is explicit through `Draw.Image(..., frameIndex, ...)`; automatic animation timing/delay handling is intentionally not implemented.
- The SharpDX guard is split between an MSBuild package-reference guard and a test that scans source/project files. This satisfies build/test enforcement without introducing a custom MSBuild file-content scanner.
- Generated solution format is the classic `.sln`, even though the .NET 11 SDK defaults to `.slnx`, because the spec explicitly names `ModernOverlay.sln`.
- `ModernOverlay` now references `ModernOverlay.Diagnostics` so EventSource diagnostics are emitted from the core lifecycle. If diagnostics must remain an entirely optional package, this should be inverted later through an internal listener/facade.
- DPI handling now uses the real HWND DPI scale for render backends and diagnostics. `WindowBounds` remains the explicit physical-pixel type per the spec, while drawing-space rectangles remain DIPs through `RectF`.

## Tradeoffs

- I used direct P/Invoke for the initial HWND lifecycle rather than wrapping every call through Vortice.Win32. This keeps the native boundary explicit and testable while still pinning `Vortice.Win32` for APIs the renderer will use.
- I used raw `nint` handles in the internal Win32 layer for the first pass. Public API exposes `WindowHandle`; a SafeHandle wrapper can be introduced once ownership and handle transfer rules are finalized.
- The initial Direct2D project contains a dependency probe instead of a renderer. This reduces risk while proving the .NET 11/Vortice package baseline before locking the transparent rendering strategy.
- DWM glass, blur-behind, and layered-alpha experiments are implemented as narrow Win32 methods instead of a broader transparency service. That keeps the spike small until the renderer backend proves which path is actually needed.
- `CA1859` is disabled because backend fields intentionally use interfaces even when the current concrete implementation is `NullRenderBackend`; preserving that abstraction is more important than the micro-optimization in this architecture.
- The Direct2D backend uses a single-threaded factory and expects all lifecycle/render calls to run on the overlay owner thread. This matches the spec's ownership model and avoids paying for Direct2D factory synchronization before a cross-thread resource story exists.
- Solid and linear-gradient brushes are currently realized lazily inside `Direct2DDrawCommandSink` and cached for the active render target by handle instance. Direct2D backend recreation clears those caches, but a later resource-system pass should move realization ownership into a backend-generation-aware resource manager so resources can be inspected and leak-tracked per handle.
- Stroke styles are also realized lazily inside `Direct2DDrawCommandSink` and cached by handle. They are disposed during backend recreation with the other Direct2D native caches, but should join the same backend-generation-aware resource manager as brushes.
- Images are realized lazily inside `Direct2DDrawCommandSink` by `ImageHandle` plus frame index and cached for the active render target. Stream-backed images are copied into the handle as encoded bytes so they can be decoded later and recreated after target changes without retaining caller-owned streams. Backend recreation disposes cached native bitmaps; like brushes/strokes, these native bitmaps should later move into the backend-generation-aware resource manager.
- Triangles are currently rendered by creating a short-lived Direct2D path geometry per draw call. That is acceptable for this parity bootstrap but should be replaced with reusable geometry realizations once `GeometryPath` descriptors and backend-generation tracking are implemented.
- `GeometryPath` currently supports `MoveTo`, `LineTo`, cubic `BezierTo`, `QuadraticBezierTo`, `ArcTo`, and `Close`. `ArcTo` maps directly to Direct2D elliptical `ArcSegment` descriptors with radius, rotation, sweep direction, and small/large arc size. Direct2D caches reusable `GeometryPath` realizations, but triangle helper geometry remains short-lived per draw call.
- Clips are currently axis-aligned Direct2D clips. That matches the public `RectF` clip API, but arbitrary path clipping would need a layer/geometry design that the current public API does not expose.
- Resource leak tracking currently records public handle id, kind, generation, allocation site, native realization count, and native realization snapshots. Frame diagnostics expose aggregate backend native realization counts, and backend recreation disposes Direct2D native caches while unregistering per-handle realization tokens.
- Target tracking now supports known HWNDs plus process ID, process name, title matching, class name, foreground-window, and custom target providers, including configurable polling intervals.
- Window-title discovery supports exact, contains, starts-with, and ends-with ordinal-insensitive matching. Regex matching is intentionally not included in the first public surface.
- Follow-target z-order is intentionally best-effort. Windows can still reorder overlays due to activation, topmost windows, secure desktops, virtual desktops, UAC prompts, or other shell behavior, and the current integration test verifies the path runs rather than proving absolute z-order.
- Minimized-target handling uses `IsIconic` polling during the existing tracking pass. This avoids a WinEvent hook for now, but it means minimize/restore is detected at the frame cadence rather than immediately.
- Positive `TargetTrackingInterval` values reduce target query overhead, but also delay move/resize/minimize/loss detection by up to that interval. The default is now 33 ms to match the spec; `TimeSpan.Zero` remains available for every-frame tracking.
- The waitable timer scheduler currently uses the public frame interval directly as the timer period. It does not yet compensate for render duration, late frames, or display-refresh changes beyond the latest-frame timing exposed through `FrameStats`.
- `DisplayDefault` refresh is resolved when `RunAsync` starts, not continuously while the overlay moves across monitors. That avoids per-frame display queries, but dynamic monitor migration should later recompute the target interval when DPI/monitor-change handling lands.
- Frame diagnostics use a 120-frame rolling window for moving-average and worst-frame durations. A frame is counted as dropped when the interval since the previous completed frame exceeds 150% of the configured target interval; unlimited rendering mode does not count dropped frames. This gives deterministic first diagnostics without trying to infer compositor-level presentation drops.
- `FrameStats` now includes rolling timing plus dropped/skipped, command, primitive, transient text layout, native resource, target HWND, and backend generation diagnostics. EventSource covers backend/device/frame/target/DPI/resource events, `Win32NativeDiagnostics` exposes the latest native failure for samples/tools, and `OverlayEventSourceLogger` bridges EventSource events to `Microsoft.Extensions.Logging`.
- `ModernOverlay.Diagnostics` suppresses `NU1510` because the logging adapter intentionally references `Microsoft.Extensions.Logging.Abstractions`; the .NET 11 SDK package-pruning analyzer otherwise flags that package as unnecessary under warning-as-error restore settings.
- `WM_DPICHANGED` handling follows Windows' suggested physical bounds and immediately resizes the render backend. The remaining DPI gap is public API semantics: `WindowBounds` is still physical pixels, not DIPs, and multi-monitor visual behavior still needs manual validation.
- Global hotkey registration is implemented but not exercised end-to-end by automated tests because OS-level hotkey availability is machine/session dependent. Current tests cover gesture helper validation; `samples/HotkeyOverlay` provides a manual validation artifact for registration, callback dispatch, input-mode toggling, and unregister-on-dispose behavior.
- Direct string text drawing still uses a large fixed layout rectangle from the requested origin. Reusable `TextLayoutHandle`s now support constrained layout size, wrapping, alignment, and trimming. Direct `Draw.Text` and `Measure.Text` calls are counted as transient text layout work; `OverlayWindowOptions.ExcessiveTextLayoutCreationThreshold` emits an EventSource warning when the per-frame count exceeds the threshold.
- Frame safety is enforced at the overlay frame boundary through a guarded command sink rather than by making `DrawContext` globally unusable outside overlays. This preserves existing headless/no-op construction while satisfying the runtime safety requirement for user-retained render frames.
- `Draw.Box` is currently a rectangle alias rather than a corner-only ESP-style box helper. If the compatibility surface needs corner boxes, that should become an explicit option or separate helper rather than changing `Box` semantics silently.
- Linear gradients currently use Direct2D's clamp extend mode and standard RGB gamma without exposing spread/gamma options. That keeps the first brush descriptor small while satisfying the spec's solid/linear-gradient brush requirement.
- Direct2D backend selection currently requires an explicit `Direct2DOverlayBackend.Register()` call from apps that reference `ModernOverlay.Direct2D`. This avoids a project cycle in the current split, but final packaging should decide whether the primary `ModernOverlay` package includes/registers the parity backend automatically.
- `PresentMode.VSync` on the Direct2D HWND backend is treated as a requested mode with an effective fallback to `BackendDefault`; `PresentMode.Immediate` maps to Direct2D immediate present options. The fallback emits one warning per backend instance instead of throwing.
- The DirectComposition spike is documented as a decision artifact rather than implemented as a backend. This matches the spec's "post-parity candidate" language, but it leaves the actual DirectComposition/DXGI implementation and benchmark comparison for a later backend-specific milestone.
- The sticky target sample references `ModernOverlay.Win32` directly so it can own and move a local target HWND. That keeps the sample deterministic and safe, but a final end-user sample set should also include a target-by-process/title example once the user experience and permission story are documented.
- The extra samples are intentionally short, build-focused console overlays. They validate API ergonomics and solution wiring, but they are not a replacement for manual visual QA across fullscreen, DPI, transparency, and multi-monitor environments. `TransparencyValidationOverlay` gives the transparency checklist a runnable artifact for the currently supported DWM glass and layered-alpha modes.
- Documentation is split into narrow topic files instead of a single generated manual. That keeps the alpha docs easy to update while the public package split and backend registration story are still unsettled.
- NuGet metadata now carries a preview warning through `PackageReleaseNotes` and uses `net11-preview` package tags. The metadata deliberately does not claim stable API or packaging status while backend registration and package split decisions remain open.
- A root `README.md` is included as `PackageReadmeFile` for all packable projects, so NuGet package creation no longer emits missing-readme warnings.
- The solution intentionally does not use virtual solution folders after Rider reported load failures for the `src` and `benchmarks` solution-folder entries. The filesystem layout still keeps source, tests, samples, and benchmarks grouped; the `.sln` now only lists real C# projects.
- Hotkey callbacks run on the Win32 owner thread. This makes operations like `overlay.InputMode = ...` safe with the existing owner-thread reentrancy behavior, but applications should keep callbacks short.
- Pointer callbacks also run on the Win32 owner thread and currently cover mouse move plus left/right/middle button press/release messages. Pointer capture, wheel, raw input, high-level gestures, and selective hit-testing are intentionally left out until the public interaction model is clearer.
- `PauseOverlay` currently pauses the overlay and then surfaces the render exception, so a caller that wants to resume after inspecting the exception must call `Resume` before starting another run loop. That matches the spec wording but may be surprising compared with a purely internal pause.
- The render-hot-path resource guard is implemented on the overlay-owned resource manager rather than globally. Standalone `OverlayResourceManager` instances still allow resource creation unless a host explicitly enters the internal render-callback scope.
- The benchmark suite now includes both no-op/public command-path benchmarks and owner-thread Direct2D render-target benchmarks. The Direct2D cases use internal backend access from the benchmark assembly to avoid global backend registration state leaking between benchmark classes.
- BenchmarkDotNet 0.15.8 does not currently recognize the .NET 11 preview runtime through its normal SDK/toolchain validator, so the benchmark harness uses the in-process emit toolchain. Revisit this after BDN or .NET 11 support catches up, because out-of-process toolchains are preferable for final performance numbers.
- Benchmark dry runs must be inspected for "Benchmarks with issues" because BenchmarkDotNet can print parser/setup errors while returning process exit code 0. The current dry run is clean except for environment metadata warnings from restricted processor/power-plan queries and expected dry-job minimum-iteration warnings.
- The root `README.md` package item is added from `Directory.Build.targets` instead of `Directory.Build.props`. This waits until project properties such as `IsPackable` are known and avoids injecting the root README as a global design-time item into every project during IDE solution load.

## Open Questions

- Confirm whether the documented DirectComposition decision is enough for this milestone, or whether you want an actual DirectComposition/DXGI backend prototype in this implementation pass.
- Confirm whether `.NET 11 preview` should remain the only TFM on `main`, or whether you want a checked-in `net10.0-windows` branch/worktree before renderer work begins.
- Confirm whether the public package split should remain four projects for alpha, or whether `ModernOverlay.Diagnostics` and `ModernOverlay.Win32` should stay internal implementation assemblies until API review.
- Confirm whether the core package should directly reference `ModernOverlay.Diagnostics` for built-in EventSource emission, or whether diagnostics should be injected/optional in final packaging.
- Confirm whether target-tracking options should store the effective resolved bounds in `OverlayWindow.Options` as they do now, or preserve the originally requested fallback bounds separately for diagnostics.
- Confirm whether the explicit `SetBoundsPixels`/`SetBoundsDips` additions are enough for alpha, or whether the older ambiguous `SetBounds(WindowBounds)` API should be renamed or hidden before package publication.
- Confirm whether title discovery also needs regex matching, or whether the current exact/contains/starts-with/ends-with modes are enough.
- Confirm whether `Draw.Box` should remain a rectangle alias or expose corner-box variants for GameOverlay.NET parity.
- Confirm whether the nine current samples are the right public set for alpha docs, or whether some should be swapped for target-by-process examples.
- Confirm whether the current topic-level docs should be expanded into package README/NuGet docs before alpha, and whether the explicit Direct2D registration should remain documented as the normal quick-start path.
- Confirm whether `UpdateLayeredWindow` should be implemented as a CPU-copy transparency fallback before feature freeze, or whether documenting it as unsupported until a later backend is acceptable for this milestone.
